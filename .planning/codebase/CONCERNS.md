# 关注点 - 技术债与问题

## 潜在问题

### 1. 硬编码的 API 端点
- 两个插件都有硬编码的基础 URL
- 缺乏基于环境的配置
- 如果 API 发生变化，会存在单点故障风险

### 2. 同步阻塞
- 轮询循环中的 `time.sleep()` 会阻塞线程
- 缺乏 async/await 支持
- 在耗时较长的视频生成过程中可能会导致 UI 冻结

### 3. 错误处理
- 某些地方使用了通用的异常捕获
- 存在静默失败的风险
- 错误消息可能会意外泄露给用户

### 4. 图像处理依赖
- Pillow 是可选的，但用于图像缩放
- 如果未安装则没有回退方案（`_ensure_min_size` 仅记录警告）

### 5. 配置持久化
- `get_params()` 在校验失败时每次调用都会写入配置
- 并发访问时可能存在竞态条件

### 6. 内存中的 Base64
- 大图完全以 Base64 形式加载到内存中
- 高分辨率图像可能导致内存问题

### 7. 缺乏单元测试
- 测试覆盖率为零
- 更改代码需要手动测试

## 安全考虑
- API 密钥以明文形式存储在配置文件中
- 请求头中使用 Bearer 令牌（标准做法但非加密）
- 代码中未见 HTTPS 证书验证逻辑

## 性能关注点
- 轮询间隔固定（非自适应）
- 缺乏连接池
- 多个视频采取顺序处理

## 可维护性
- 文件体积较大（每个超过 800 行）
- 插件之间的重复模式可以提取共享
- 中文注释可能会限制外部贡献
