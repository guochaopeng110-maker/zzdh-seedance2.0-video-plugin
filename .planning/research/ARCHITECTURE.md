### 组件职责

| 组件 | 职责 | 典型实现 |
|-----------|----------------|------------------------|
| 插件契约适配器 | 保持面向宿主的 API 稳定 (`get_info/get_params/generate`) | 不含供应商特定分支的精简函数 |
| 编排器 | 执行端到端生命周期和重试 | 单个状态机风格的协调器 |
| 参数/能力层 | 将宿主参数转换为供应商安全的数值 | 清洗器 + 能力映射 (分辨率/比例/时长/模式) |
| 供应商适配器 | 隔离 ZLHub 请求/响应模式 | `build_create_payload`, `parse_status`, `extract_video_url` |
| 轮询引擎 | 控制状态检查/超时/退避 | 带有明确终端状态的有限循环 |
| 产物下载器 | 下载最终的 mp4 并持久化路径 | 优先 URL 下载，失败时回退到认证接口 |
| 错误/进度转换器 | 规范化错误和 UI 进度信号 | `PLUGIN_ERROR:::` 前缀 + 回调映射 |
| 配置/日志层 | 持久化插件配置和可观测性 | `plugin_utils` + 可选的 SQLite 日志 |

## 建议的项目结构

```text
video_plugin_zlhub_seedance/
├── main.py                       # 仅限宿主契约: get_info/get_params/generate
├── core/
│   ├── orchestrator.py           # 提交/轮询/下载工作流
│   ├── params.py                 # 默认值 + 清洗 + 能力规范化
│   ├── errors.py                 # 统一的插件错误类型/前缀转换
│   └── progress.py               # 宿主回调事件映射
├── provider/
│   ├── zlhub_seedance_adapter.py # ZLHub 创建/查询模式 + 状态解析
│   └── model_caps.py             # 允许的选项与约束表
├── infra/
│   ├── http_client.py            # requests 封装，超时/重试策略
│   ├── polling.py                # 轮询策略和停止条件
│   ├── downloader.py             # 文件下载 + 命名 + 回退
│   └── storage.py                # 配置/任务日志持久化
└── tests/
    ├── test_params.py
    ├── test_adapter_mapping.py
    └── test_status_transitions.py
```

### 结构合理性说明

- **`main.py` 边界:** 保持宿主集成不可变；所有变更都发生在该边界之后。
- **`provider/` 隔离:** 防止 ZLHub 的模式变更影响到编排层。
- **`core/` 编排:** 集中生命周期控制，使重试/轮询逻辑保持一致。
- **`infra/` 分离:** 避免在 Payload 逻辑中混入传输和业务规则。

## 架构模式

### 模式 1: 稳定的入口点，可替换的内部实现
**内容:** 面向宿主的 API 是固定的；内部实现是模块化且可交换的。
**使用时机:** 宿主无法更改（本里程碑的硬边界）。
**权衡:** 前期结构成本略高，但未来的供应商更换会安全得多。

### 模式 2: 适配器映射前的能力规范化
**内容:** 先规范化用户/宿主输入，然后再映射到供应商的 Payload 中。
**使用时机:** 供应商具有严格的枚举/范围规则（Seedance 对比例/分辨率/时长有限制）。
**权衡:** 代码路径更多，但 API 失败更少，且错误消息更清晰。

### 模式 3: 显式的异步任务状态机
**内容:** 将创建/轮询/下载视为具有终端结果的显式状态。
**使用时机:** API 返回任务 ID 并需要轮询。
**权衡:** 稍微繁琐，但能防止隐藏的无限循环和模糊的失败。

## 数据流

### 请求流程

```text
[宿主 UI 触发]
    ↓
generate(context)
    ↓
参数规范化 (模型/模式/比例/时长/媒体约束)
    ↓
供应商适配器构建创建 Payload
    ↓
POST /tasks (创建)
    ↓ 返回 task_id
轮询引擎
    ↓ 重复 GET /tasks/{id}
[status=running] ──循环──> 再次轮询
[status=completed]
    ↓
提取下载 URL (或内容端点回退)
    ↓
下载器将 mp4 保存到输出目录
    ↓
返回 [output_path] 给宿主
```

## 建议的构建顺序 (用于路线图阶段)

1. **契约外壳先行:** 创建具有稳定 `get_info/get_params/generate` 签名和最小传递逻辑的 `main.py`。
2. **参数/能力层:** 在进行任何网络逻辑之前，实现来自 Seedance 文档的约束。
3. **供应商适配器 (创建 + 查询解析器):** 使用单元测试构建 Payload/状态提取。
4. **编排器 + 轮询状态机:** 连接提交/轮询的终端状态和超时边界。
5. **下载器 + 输出命名:** 完成产物检索和认证回退。
6. **错误/进度转换:** 统一面向用户的错误和回调消息。
7. **任务日志/重试调优 (硬化):** 增加运维可见性和韧性。

## 反模式

### 反模式 1: 供应商模式泄露到宿主边界
**做法:** 将原始 ZLHub 字段一直传递到 UI/宿主。
**错误原因:** 宿主与单一供应商耦合；未来的插件更换需要修改宿主。
**应采取的做法:** 将供应商特定字段保留在 `provider/` 内部，仅向外暴露规范化的插件级参数。

### 反模式 2: 庞大的 `generate()`
**做法:** 将清洗/构建/轮询/下载/错误处理全部放在一个大函数中。
**错误原因:** 难以测试；变更会在无关行为中产生回归。
**应采取的做法:** 保持 `generate()` 精简，并委托给编排器和适配器模块。

### 反模式 3: 无限制轮询
**做法:** 无限或近乎无限的轮询，缺乏终端保护。
**错误原因:** 资源浪费，且在供应商发生事故时 UX 极差。
**应采取的做法:** 明确最大尝试次数、超时时间和终端状态映射 (`FAILED/TIMEOUT`)。
