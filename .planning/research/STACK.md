# 技术栈研究

**领域:** 用于第三方视频生成 API 集成的 Python 桌面应用插件 (提交 → 轮询 → 下载)
**研究日期:** 2026-03-30
**置信度:** 高

## 建议的技术栈

### 核心技术

| 技术 | 版本 | 用途 | 推荐理由 |
|------------|---------|---------|-----------------|
| Python | **3.12.x** (目标), 3.13 兼容 | 插件运行时 | 2026 年桌面插件生态系统的最佳兼容性/安全性权衡。 **置信度: 高** |
| requests | **2.33.0** | 用于提交/轮询/下载的同步 HTTP | 对于宿主加载的桌面插件，同步 I/O 比引入事件循环复杂度更简单、更安全。 **置信度: 高** |
| pydantic | **2.12.5** | 请求/响应模式校验 | 第三方视频 API 经常变动；严格的模型可以及早捕获契约偏离。 **置信度: 高** |
| tenacity | **9.1.4** | 重试/退避策略 | 轮询过程中预期会出现短暂的网络/API 失败。Tenacity 提供了可组合的重试策略。 **置信度: 高** |
| structlog | **25.5.0** | 结构化日志 | 插件的可观测性需要机器可读的日志。 **置信度: 高** |

### 开发工具

| 工具 | 用途 | 备注 |
|------|---------|-------|
| ruff | Lint + 导入排序 + 静态检查 | 作为单一的快速 Linter 以减少插件回归。 |
| mypy | 静态类型检查 | 校验请求/响应的 DTO 边界。 |
| pytest | 单元/集成测试 | 对于序列化器/校验器和轮询状态转换是强制性的。 |

## 替代方案考虑

| 推荐方案 | 替代方案 | 何时使用替代方案 |
|-------------|-------------|-------------------------|
| requests 2.33.0 | httpx 0.28.1 | 仅当你需要同时支持同步+异步的客户端时。 |
| tenacity 9.1.4 | 手写重试循环 | 仅适用于琐碎的原型开发。 |

## 不建议使用的项

| 避免使用 | 原因 | 替代建议 |
|-------|-----|-------------|
| 不带显式 `timeout` 的 `requests` | 可能导致无限期挂起；在桌面插件 UX 中是致命的。 | 始终设置连接/读取超时。 |
| 默认使用异步优先栈 (`aiohttp`) | 增加集成风险和调试成本。 | 从同步方式开始，必要时采用工作线程模型。 |
| 裸 `print` 日志 | 无法支持按任务 ID 进行故障排查。 | 使用 `structlog`。 |
