# Phase 08.2 Research: 固定参数策略与安全治理（URL/TOS/AKSK来源）

**Date:** 2026-04-24
**Phase:** 08.2
**Status:** Complete

## Research Question

在不改变宿主调用契约前提下，如何为 `video_plugin_zlhub_seedance_V2` 建立“参数来源治理”基线，确保：
1. 视频任务/审核接口 URL 来源可追溯、可验证且不可被错误覆盖；
2. TOS 关键参数（endpoint/region/bucket）来源统一；
3. AK/SK 不再由 UI 或不受控输入散落进入运行时。

## Findings

### 1. 当前实现的参数来源存在混合路径
- `base_url` 仍由 UI 暴露并持久化，运行时再拼接 `task_create_url/task_query_url`。
- `audit_access_token` 来自 UI，合理；但 URL 类固定常量与可输入配置并存，容易出现“看起来可配、实际被覆盖”的歧义。
- `tos_ak/tos_sk` 仍在参数表中读取，存在来源不统一与误配置风险。

### 2. requires2 与 TOS 文档已经给出强约束
- 视频接口与审核接口已有固定官方路径，适合作为“默认且优先”的策略基线。
- TOS 对接文档已固定 endpoint/region/bucket，AK/SK 需要受控来源（环境变量/受限配置），不应在 UI 明文编辑。

### 3. 最低风险治理策略
- 将“固定协议参数”与“业务可调参数”分层：
  - 固定层：task/audit/tos endpoint 与 bucket/region（代码常量 + 配置校验）
  - 可调层：api_key、audit_access_token、模型参数、超时参数
- 对来源增加可观测性：启动时记录“来源标签”而非明文值（例如 `source=constant/env`）。

## Implementation Implications

1. 参数治理重构
- 新增统一策略函数（如 `_resolve_runtime_sources`），按优先级收敛 URL/TOS 来源。
- 移除或弃用 `base_url` 作为用户可配项，避免查询/创建 URL 漂移。

2. UI 面收敛
- `ui/index.html` 下线 `BaseUrl` 输入控件。
- 保留 `audit_access_token`、业务参数；明确“固定接口地址由插件内置策略维护”。

3. AK/SK 来源治理
- 运行时优先从环境变量读取（如 `ZLHUB_TOS_AK`/`ZLHUB_TOS_SK`），无值时按本 phase 固定策略兜底。
- 日志与任务记录中仅记录是否存在，不记录明文。

4. 防漂移校验
- 增加自动化检查：扫描 V2 代码中视频/审核 endpoint、TOS endpoint 是否与文档一致。

## Validation Architecture

### Unit-level
- `_sanitize_params` 不再接受 UI `base_url` 改写核心 endpoint。
- 来源解析函数对 `constant/env/user` 的优先级行为可测。
- AK/SK 缺失时错误路径包含 `PLUGIN_ERROR:::`。

### Integration-level
- 仿真人审核链路仍只接收公网 URL；调用的 submit/query endpoint 固定。
- 视频 create/query 走 requires2 固定路径且包含 `X-Trace-ID`。

### Regression-level
- 现有 `audit_test_only`、轮询与下载主链路行为不退化。
- UI 现有模型/分辨率/时长参数保存能力不受参数治理改造影响。

## Risks and Mitigations

- 风险：移除 `base_url` 影响历史配置兼容。
  - 缓解：保留旧字段读取但标记 deprecated，运行时忽略并记录一次迁移日志。

- 风险：AK/SK 来源切换导致线上缺失。
  - 缓解：提供来源探针日志（仅布尔）+ 明确错误指引。

- 风险：文档更新后常量漂移。
  - 缓解：增加关键字检测脚本，执行阶段强制运行。
